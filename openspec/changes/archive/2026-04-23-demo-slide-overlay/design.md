## Context

Demo mode (`codemoo demo`) runs a sequence of `ChatApp` instances back-to-back. Each bot gets a fresh chat session. Currently the transition is silent: the user presses Ctrl-N and the next chat immediately opens. There is no moment for the presenter to orient the audience.

The new slide screen must feel like an overlay on the chat (not a separate app restart), load LLM content without blocking the presenter, and carry awareness of the full bot sequence for the Agenda.

## Goals / Non-Goals

**Goals:**
- Show a `ModalScreen` before each bot session with Agenda + bot introduction
- LLM-generated "what's new" explanation loaded async, never blocks the presenter
- Bot session filtering: `--start rune` produces `[Rune, Ash, Loom]`, earlier bots don't exist
- Keyboard: Enter and Escape dismiss in addition to the OK button

**Non-Goals:**
- Not animated or transitions between slides
- Not interactive beyond dismiss — no navigation between bots from the slide
- No changes to `make_bots()` or any core bot classes
- LLM content is not cached; it varies per run by design

## Decisions

### ModalScreen over a standalone SlideApp

Alternatives: run a separate Textual `App` before each `ChatApp` (like `SelectionApp`), or use a layered widget inside `ChatApp`.

Decision: **`ModalScreen` pushed via `ChatApp.on_mount()`**. This gives the overlay-on-top-of-chat visual without a cold app restart, and Textual's built-in modal dimming works for free. The chat layout composes first; the modal blocks interaction until dismissed.

### DemoContext dataclass for threading

`ChatApp` currently takes `demo_position: tuple[int, int] | None`. The slide needs `all_bots`, `prev_bot`, and `backend` as well.

Decision: **Single `DemoContext` dataclass** (`all_bots`, `prev_bot`, `backend`, `position`). Replace the `demo_position` param with `demo_context: DemoContext | None`. Avoids three new constructor params; groups all demo-specific state cleanly.

`_setup()` is updated to return the backend so `demo()` can include it in `DemoContext`.

### Bot list filtering in `demo()`, not `make_bots()`

Filtering is presentation logic, not construction logic. `make_bots()` stays pure.

Decision: **Slice in `demo()`** — `demo_bots = available[index:]`. Position numbers and the Agenda use only `demo_bots`. If `--start rune`, earlier bots don't appear anywhere.

### Inline source over tool use for LLM content

The "what's new" explanation could be generated by an agent that reads its own source files via `read_file`. Alternatives: inline source in the prompt, or trust the LLM without source context.

Decision: **Inline source** in the prompt. Bot files are 10–35 lines each; inlining is fast and deterministic input. Tool use adds latency and unpredictable paths for a live demo. The prompt includes both bot source files (current and previous) plus a runtime tools summary when the bot has a `.tools` attribute.

### Per-type source file mapping in `slides_data.py`

`ToolBot`, `FileBot`, and `ShellBot` are thin `GeneralToolBot` subclasses: their own file is ~4 lines (just `instructions`). Without `general_tool_bot.py`, the LLM explanation would be vacuous.

Decision: **`_BOT_SOURCES` dict** in `slides_data.py` mapping `type → list[str]`. GeneralToolBot subclasses include `["<own_file>.py", "general_tool_bot.py"]`. This is curated, so the mapping is explicit and not derived from class hierarchy at runtime.

### Async worker with always-enabled OK button

The LLM call takes 1–5 seconds. Blocking the slide until it completes would stall the presenter.

Decision: **Worker fires on `SlideScreen.on_mount()`**; content widget shows "Generating…" until the worker posts a result. The OK button is always enabled. If the presenter dismisses early, the worker is cancelled naturally by Textual when the screen unmounts.

## Risks / Trade-offs

- **LLM variability**: Content differs each run. Acceptable by design — the explanation is educational, not scripted.
  → Mitigation: prompt explicitly constrains length ("5–8 lines") and focus ("code-focused, key concept only").

- **ModalScreen dimming**: If Textual's default modal dim is too dark, the agenda column may be hard to read.
  → Mitigation: style the modal background explicitly in `chat.tcss`.

- **Worker cancellation on fast dismiss**: If the presenter presses OK before the LLM responds, the worker is cancelled mid-flight. No cleanup needed since there's no persistent state, but the backend may have already incurred the API call.
  → Accepted trade-off; no mitigation needed for demo use.

- **`_setup()` signature change**: All callers of `_setup()` (`chat`, `select`, `demo`) will receive the backend in the return tuple. The `chat` and `select` commands don't use it and will ignore it.
  → Low risk; the change is local to `tui.py`.
